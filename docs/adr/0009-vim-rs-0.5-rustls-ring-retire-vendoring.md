# 0009 — Adopt vim_rs 0.5 (rustls/ring, BYOC); retire the vendoring pipeline

- **Status:** Accepted
- **Date:** 2026-06-03
- **Deciders:** Erick Bourgeois
- **Related:** ADR-0008 (BYOC for the vSphere HTTP client) — this resolves its
  open "build-side" caveat; ADR-0006 (release & supply-chain pipeline);
  `crates/banlieue-provider-vsphere/src/client/vim.rs`; the vendoring pipeline
  being removed (`patches/vim_rs.patch`, Makefile `vendor-vim-rs`,
  `.github/actions/vendor-vim-rs`, `third_party/vim_rs`); the libssl scaffolding
  being reverted (`Dockerfile`, `Dockerfile.chainguard`, `Cross.toml`);
  upstream [noclue/vim_rs#37](https://github.com/noclue/vim_rs/issues/37).

## Context

ADR-0008 adopted Bring-Your-Own-Client for the vSphere transport but left one
concern explicitly unresolved: **which TLS backend is compiled in.** Unpatched
`vim_rs` 0.4.x declared `reqwest = "0.12"` with default features, pulling
`default-tls → native-tls → OpenSSL`. Because Cargo features are additive, a
downstream crate could not subtract OpenSSL by choosing rustls for itself. We
worked around this with a one-line vendored patch (`patches/vim_rs.patch`)
flipping `vim_rs`'s reqwest to `default-features = false` + rustls, applied to a
git checkout at build time by `make vendor-vim-rs` and wired into every cargo
target and CI job. That patch carried a substantial tail: a pinned commit, a
`[patch.crates-io]` redirect, a vendored `third_party/vim_rs` checkout, a stamp
file to avoid rebuilds, a CI composite action — plus, downstream of the OpenSSL
linkage, a libssl stage in both Dockerfiles and a `libssl-dev` install in
`Cross.toml` for macOS→Linux cross-compilation.

The upstream author shipped the fix as **vim_rs 0.5.0** (on crates.io,
2026-06). Verified facts about 0.5.0:

- **reqwest 0.13, rustls by default — no OpenSSL.** `reqwest = { version =
  "0.13", default-features = false }`; the crate's `default-client` feature
  pulls `reqwest/default` (rustls). The OpenSSL linkage that justified the patch
  is gone at the source.
- **BYOC is a first-class mode.** A new default-on `default-client` feature wraps
  turnkey setup. With it **off**, `ClientBuilder::new(server, client)` *requires*
  a caller-supplied `reqwest::Client` and the `.http_client()` builder method
  does **not** exist; with it on, `ClientBuilder::new(server)` auto-creates one
  and `.http_client()` / `.insecure()` remain available. This is exactly the
  design the author and ADR-0008 converged on.
- **`vcsim_compat` retained** (`vcsim_compat = ["xml"]`), so local-dev/vcsim is
  unaffected.

Two consequences of the bump need decisions:

1. **reqwest major bump 0.12 → 0.13.** banlieue holds a *direct* reqwest dep (for
   the BYOC client, ADR-0008). It must move to 0.13 so the injected
   `reqwest::Client` type matches the one `vim_rs` 0.5 expects — otherwise two
   reqwest majors coexist and the types don't unify. The rustls feature was also
   renamed: 0.12's `rustls-tls-native-roots` no longer exists; 0.13 exposes
   `rustls` (aws-lc-rs provider + platform verifier) and `rustls-no-provider`.

2. **Crypto provider choice.** reqwest 0.13's `rustls` feature defaults to the
   **aws-lc-rs** provider (a C/asm build dependency). The rest of the workspace
   already standardises on **ring**: kube pulls `rustls v0.23` on the **ring**
   provider (`ring v0.17 → rustls → hyper-rustls → kube-client`), and aws-lc-rs
   is absent from the tree. Taking reqwest's default `rustls` would add a second
   crypto provider and reintroduce the cross-compile friction we just removed.

## Decision

**Migrate to `vim_rs = "0.5"` from crates.io and delete the vendoring pipeline.
Move the direct reqwest dep to 0.13 with the `ring` crypto provider. Use
`vim_rs` in BYOC mode (`default-client` off). Revert the libssl Docker/cross
scaffolding now that OpenSSL is gone from the graph.**

Concretely:

- **Dependency.** `vim_rs = { version = "0.5", default-features = false }` in
  `[workspace.dependencies]` — plain crates.io, no exact pin. Delete the
  `[patch.crates-io]` block. The provider crate enables only the features it
  needs (`vcsim_compat` behind its own `vcsim` feature); `default-client` stays
  off so BYOC is mandatory and no turnkey reqwest client is ever built by
  `vim_rs`.
- **reqwest.** `reqwest = { version = "0.13", default-features = false, features
  = ["rustls-no-provider", "charset", "http2"] }`. `rustls-no-provider` links
  rustls without baking in a crypto provider; we supply **ring**.
- **One crypto provider, installed once.** reqwest 0.13 resolves the rustls
  provider via `CryptoProvider::get_default()`, and with `rustls-no-provider`
  **panics (`"No provider set"`) if none is installed.** So the binary installs
  `rustls::crypto::ring::default_provider()` as the process default once at
  startup, before any TLS use. This is shared with kube (same ring provider),
  keeping the workspace on a single crypto stack.
- **BYOC call site.** `ClientBuilder::new(&endpoint, http)` (2-arg form);
  `.http_client(http)` is removed in `default-client`-off mode. banlieue still
  builds the `reqwest::Client` itself (CA bundle, insecure flag) per ADR-0008.
- **Retire the pipeline.** Remove `make vendor-vim-rs` (+ all prerequisites and
  `VIM_RS_*` vars + stamp), `.github/actions/vendor-vim-rs` and the CI steps that
  call it, `patches/vim_rs.patch` + `patches/README.md`, the `third_party/vim_rs`
  checkout and its `.gitignore` entry.
- **libssl scaffolding.** Already removed when the rustls patch first landed
  (the Dockerfiles are single-stage rustls/no-OpenSSL COPYs and `Cross.toml` is
  gone), so there is nothing left to revert here — noted for completeness so the
  "OpenSSL-free" end state is documented in one place.

## Consequences

**Positive**

- **OpenSSL fully gone, no patch.** A plain crates.io dependency replaces the
  entire vendor/patch/stamp/CI apparatus. Far less to maintain and explain.
- **Single crypto provider.** ring serves both kube and the vSphere client; no
  aws-lc-rs, no dual TLS stacks, no new cross-compile toolchain requirement.
- **Simpler images and cross builds.** Dropping the libssl stage shrinks both
  Dockerfiles and removes `libssl-dev` from `Cross.toml`; macOS→aarch64 builds
  no longer need a Linux libssl sysroot.
- **Closes ADR-0008's caveat.** The "does not retire the vendoring pipeline"
  trade-off recorded there is now resolved by an upstream release.

**Negative / trade-offs**

- **Provider must be installed or TLS panics.** With `rustls-no-provider`,
  forgetting `ring::default_provider()` install yields a runtime
  `"No provider set"` panic on first TLS use, not a compile error. Mitigated by
  installing it at the single binary entry point and asserting it in a startup
  test.
- **reqwest 0.13 is a major bump.** A direct-dep API surface change; our usage is
  small (`Client::builder`, `from_pem_bundle`, `add_root_certificate`,
  `user_agent`, `danger_accept_invalid_*`), all verified present in 0.13. SOAP/
  ESXi consumers would also need `reqwest/cookies` — N/A for us (JSON/vCenter
  path; `cookie_store` is only set on vim_rs's SOAP builder).
- **Feature coupling to keep aligned.** banlieue's direct reqwest features must
  stay rustls/ring-consistent with whatever `vim_rs` resolves; if a future
  vim_rs bump changes its reqwest, revisit. Documented at the dep site.

## Alternatives considered

- **Stay on 0.4.4 + the patch.** Keeps a fork-shaped maintenance burden for a
  problem upstream has now solved. Rejected.
- **reqwest default `rustls` (aws-lc-rs).** One fewer feature flag, but adds a
  second crypto provider to a tree already standardised on ring and reintroduces
  the C/asm cross-compile dependency we removed. Rejected in favour of
  `rustls-no-provider` + ring.
- **Keep `default-client` on and pass our client via `.http_client()`.** Works,
  but leaves `vim_rs` able to construct a turnkey reqwest client (pulling
  `reqwest/default`) we never use — re-adding default reqwest features to the
  graph. Turning `default-client` off makes BYOC the only path and keeps the
  feature surface minimal. Rejected.
- **Defer the Docker/cross libssl revert to a follow-up.** Smaller diff, but
  leaves dead OpenSSL scaffolding that reads as load-bearing. Done here so the
  end state is coherent and the "why" is captured in one place.
