## Key Learnings

- **Project:** banlieue
- **Description:** A Kubernetes-native abstract virtualization API. Users create `VirtualMachine` CRs; controllers schedule them onto vSphere / Proxmox / libvirt backends via provider-specific infrastructure CRDs that satisfy the CAPI v1beta2 InfraMachine contract.
- **Architecture invariant:** No RPC between main controller and providers — communication is **CRD-only**. If a design wants HTTP/gRPC between them, it's wrong.
- **Workspace layout:** Cargo workspace. Phase 0 (done) ships `crates/banlieue-api` with two API groups: `banlieue.io/v1alpha1` (`Provider`, `VMClass`, `VMImage`, `VirtualMachine`) and `infrastructure.banlieue.io/v1alpha1` (`VSphereMachine`, `VSphereMachineTemplate`). Phase 1+ adds `banlieue-controller`, `banlieue-provider-sdk`, and `banlieue-provider-{vsphere,proxmox,libvirt}` per the out-of-repo roadmap at `~/dev/roadmaps/banlieue/`.
- **Single binary, library role crates (ADR-0004):** The only executable is `crates/banlieue` (the `banlieue` binary). It just parses a clap subcommand tree and dispatches into the role crates. `banlieue-controller` and every `banlieue-provider-*` are **libraries** (no `main.rs`/`[[bin]]`) that each export a public `Cli` (`#[derive(clap::Args)]`) and `pub async fn run(cli) -> anyhow::Result<()>`. Providers are gated by per-provider features on the `banlieue` crate (`default = ["vsphere"]`; `vcsim` passes through to `banlieue-provider-vsphere/vcsim`). The Deployment selects the role via container `args`.
- **clap pattern for role `Cli`s:** role `Cli` structs derive `clap::Args` (not `Parser`) so they embed as subcommand payloads (`Command::Controller(banlieue_controller::Cli)`). To parse one standalone in a unit test, wrap it: `#[derive(Parser)] struct Wrapper { #[command(flatten)] cli: Cli }` then `Wrapper::parse_from([...]).cli`. See `app_tests.rs` in the controller / vsphere crates.
- **Shared process bootstrap lives in `banlieue-provider-sdk::bootstrap`:** `init_tracing(format, level, extra_directives)`, `serve_health(port)`, `shutdown_signal()`. Per-crate log directives passed as `extra` (`["kube=warn"]` for the controller, `["kube=warn","vim_rs=warn"]` for vsphere). Don't re-implement these in role crates.
- **CRD generation:** `cargo run -p banlieue-api --bin crdgen --features crdgen` — the `crdgen` binary is feature-gated, so the flag is required. `serde_yaml` is also gated on the `crdgen` feature via `crdgen = ["dep:serde_yaml"]`.
- **License:** Apache-2.0 (workspace-wide, set in `[workspace.package]`).
- **Roadmap filename convention:** UPPERCASE with hyphens, numeric prefix (`00-OVERVIEW.md`, `10-PHASE-1A-CONTROLLER-AND-SDK.md`). ADRs use lowercase-hyphen.
- **k8s-openapi needs the `schemars` feature** for `Condition`, `Time`, and other meta/v1 types to implement `JsonSchema`. Without it, any field like `Vec<Condition>` or `Option<Time>` in a `JsonSchema`-deriving struct fails to compile. Pinned in workspace `[workspace.dependencies]`.
- **vim_rs is patched to rustls (no fork, no in-tree vendoring).** `vim_rs`'s reqwest defaulted to native-tls → OpenSSL on Linux (the lone OpenSSL puller; kube is already rustls). Fix: `[patch.crates-io] vim_rs = { path = "third_party/vim_rs/vim_rs" }` + `make vendor-vim-rs` clones `noclue/vim_rs` at `VIM_RS_REF` (a commit SHA — 0.4.4 has `vcsim_compat` but was never git-tagged; only `v0.4.3` exists) into gitignored `third_party/vim_rs/`, applies `patches/vim_rs.patch` (one hunk: reqwest → `default-features=false, features=["rustls-tls-native-roots","charset","http2"]`). Cargo dep pinned `=0.4.4` exact so the patch resolves. **Result: workspace is OpenSSL-free (`openssl-sys: 0, rustls: 1` in Cargo.lock — like bindy/5-spot); cross-compiles from macOS with the plain gcc cross-toolchain, no `cross`/libssl/sysroot.** Every cargo-running make target depends on `vendor-vim-rs`; a bare `cargo` needs `make vendor-vim-rs` first (checkout is gitignored). Refresh per `patches/README.md`; retire (drop `[patch]`, bump version) once upstreamed.
- **MSRV is Rust 1.88, driven by `kube 3.x`.** `kube` 3.0.0+ declares `rust-version = 1.88` (kube 2.0.1 needs 1.85). The workspace pins `kube = "3"` (resolves 3.1.0), so `[workspace.package] rust-version` must be `1.88`. The repo uses `resolver = "2"`, which is **not** MSRV-aware, so cargo silently resolves deps exceeding the declared MSRV — but `cargo upgrade` *is* MSRV-aware and flags `kube` as "incompatible" if the declared MSRV is below what kube 3.x needs. If MSRV and kube disagree again, fix the MSRV (or move to `resolver = "3"`); don't downgrade kube to satisfy a stale MSRV. (2026-05-31)
- **Test file colocation requires `#[path]`** because `src/foo.rs` is a leaf file-module. Pattern: at the bottom of `src/foo.rs` put `#[cfg(test)] #[path = "foo_tests.rs"] mod foo_tests;`, then put a `mod tests { use super::super::*; ... }` inside `src/foo_tests.rs`. Without `#[path]`, rustc looks for `src/foo/foo_tests.rs`. Test files do **not** need to re-import `crate::common::*` — `use super::super::*` already pulls in everything in scope in the parent module (child modules see private `use` imports from the parent).
- **CRD API reference is generated:** `docs/src/reference/api.md` is produced by the `crddoc` binary (`crates/banlieue-api/src/bin/crddoc.rs`, logic in `src/crddoc.rs`), NOT hand-edited. It walks each CRD's `openAPIV3Schema` into Markdown field tables. `make api-docs` regenerates it; `make crds` runs `api-docs` as its last step, and `make docs` depends on `api-docs` (so the docs.yaml CI workflow regenerates the reference on every docs build — the Rust toolchain is installed in that job). Any CRD change refreshes both the YAML and the reference page. To change the reference content, edit the rustdoc on the Rust types and regenerate. mkdocs nav entry lives under Reference.
- **kube-derive CRD descriptions:** the doc comment on a `*Spec` struct becomes the `spec` *property* description, NOT the schema root. kube-derive hard-codes the root `openAPIV3Schema.description` to "Auto-generated derived type for `<T>` via `CustomResource`" and there's no macro knob to override it. To get authored "what is this resource" text on a bare `kubectl explain <kind>`, post-process the generated CRD: see `crates/banlieue-api/src/crdgen_support.rs::promote_spec_description` (copies the spec description up to the root), applied via `prepared()` in the crdgen bin. Field/nested-struct `///` comments DO flow through normally. Module-level `//!` docs are NOT captured.
- **Never put a `_tests.rs` file directly in `src/bin/`** — cargo auto-discovers every `src/bin/*.rs` as its own binary, so a test-only file there compiles as a standalone bin and `use super::super::*` fails. Put the logic + its `_tests.rs` in the library (feature-gated if needed) and call it from the bin. This is why crdgen's post-processing lives in `src/crdgen_support.rs`, not `src/bin/`.
- **CAPI contract label is emitted by crdgen, NOT kustomize (2026-05-31, ADR-0005):** kube-derive cannot set CRD `metadata.labels`, so the CAPI discovery label `cluster.x-k8s.io/v1beta2: <served versions>` is added in `crdgen_support::add_capi_contract_label` (applied by `prepared()`). Scoped by `crd.spec.group == "infrastructure.banlieue.io"` — only infra CRDs (VSphereMachine/Template, VSphereCluster) get it; `banlieue.io` CRDs (Provider/VirtualMachine/VMClass/VMImage) do NOT (not CAPI contract surfaces). Value = comma-joined served version names. There is NO kustomize overlay (earlier docstrings/ADR-0002 said "via kustomize" — superseded). The separate `cluster.x-k8s.io/aggregate-to-manager: "true"` label is on the controller *ClusterRole* (RBAC), not the CRD.
- **Release & supply-chain pipeline lives in `.github/workflows/build.yaml` (ADR-0006, 2026-05-31, modeled on `~/dev/5-spot`):** the single `banlieue` binary is the core released artifact. Jobs: `build` (per-arch `banlieue-linux-{amd64,arm64}` + `make sbom` CycloneDX) → `docker` (matrix Chainguard+Distroless, multi-arch buildx, Cosign keyless sign by digest, BuildKit sbom+provenance, anchore image SBOM) → `attest` (GitHub build-provenance) → `build-vex` (vexctl merge `.vex/*.json`, Cosign `--type openvex` attest; empty-VEX-safe) → `grype` (`--vex` → SARIF) → `sign-artifacts` (tarball+Cosign+attest) → `generate-provenance-subjects` + `slsa-provenance` (slsa-github-generator `@v2.1.0`, MUST be a tag not SHA) → `upload-release-assets`. **Images build on PRs but push/sign/attest/scan only on push-to-main+release** (fork-PR safety). `make` targets CI shells to: `sbom`, `vexctl-install`, `crds`. `.vex/` holds curated OpenVEX (`*.json`, merged) + `.affected-functions.json` (dot-prefixed scaffold). Composite `.github/actions/prepare-docker-binaries` stages artifacts at `binaries/<arch>/banlieue` for the Dockerfiles. **Auto-VEX is implemented** in `crates/banlieue-vex` (ported from 5-spot): `auto-vex-presence` (SBOM-absence → `component_not_present`) + `auto-vex-reachability` (binary `nm -D` symbol-import absence vs `.vex/.affected-functions.json` → `vulnerable_code_not_in_execute_path`). CI jobs `grype-triage` → `auto-vex-presence` + `auto-vex-reachability` → `build-vex` (merges curated `.vex/*.json` + both auto docs). Local mirrors: `make vex-auto-presence` / `vex-auto-reachability`. Validate workflows with `actionlint` (installed). **DON'T re-stage the auto-vex binaries — the maintainer confirmed they're built/used in 5-spot.**

- **Makefile env-config knobs must be CLI-overridable:** use `VAR ?= default` and reference `$(VAR)` in recipes — never hardcode the value inline (e.g. `RUST_LOG=info,kube=warn cargo run`), because that clobbers any value the user exports on the CLI. `?=` yields to an environment-provided value, so `RUST_LOG=debug make run-local` then works. - **kube-derive's CRD flattener cannot handle serde-tagged enums** (the actual root cause behind the original buglog bug-006). Each variant subschema of a `#[serde(tag = "x")]` enum naturally fixes `x` to a different enum value, but kube-core demands identical schemas for properties shared across subschemas. The CRD-generation panic is therefore fundamental to tagged enums, not a metadata bug. **The correct Kubernetes pattern is a flat struct** with a discriminator-string field plus optional sibling fields per variant. This is how `IpamSpec` was redesigned on 2026-05-25 (bug-006 fixed). Future learning: do not introduce `#[serde(tag = ...)]` enums on any type that needs to land in a CRD.
- **k8s-openapi 0.27 uses `jiff::Timestamp` for `Time`, not `chrono`.** Code that referenced `k8s_openapi::chrono::DateTime<Utc>` fails to compile. To produce a `Time` in tests without a direct jiff dep, deserialize an RFC3339 string: `serde_json::from_str::<Time>(&format!("\"{}\"", rfc3339)).unwrap()` — this works because `Time` is `#[derive(Deserialize)]`.

Root-level: `Dockerfile`, `Tiltfile`, `justfile`, `charts/`, `yaml/`. **banlieue is multi-crate by design** (controller + provider SDK + 3 providers) so we adopt their *per-crate conventions*, not their single-crate layout.

- **Scheduler shape (Phase 1A iter 2, 2026-05-26):** pure function `crates/banlieue-controller/src/reconciler/scheduler::schedule(vm, class, image, providers, existing_vms) → Result<Decision, ScheduleError>`. Owned `Decision` carries `provider_name`, `provider_namespace`, `provider_class`, `failure_domain_name`, `resolved_storage` / `resolved_networks` (both `Vec<ResolvedResource>`), `failure_domain_raw` (BTreeMap from `attributes.raw` — opaque, consumed by provider-specific infra builders), and `failure_domain_labels`. Filter chain order matters: providerSelector → failureDomainSelector → image readiness → storage classes → network classes → features → firmware → required anti-affinity. Tie-break: alphabetical by `(provider_name, fd_name)`. `Decision::to_scheduled_placement(now)` converts to the API type.
- **Backend ID resolution rule (2026-05-26):** when a Provider declares `StorageClassMapping { name: "gold", target: { datastore: "ds-1" } }` (or similar), the scheduler resolves "gold" to a single backend_id string by taking **the first value of the target BTreeMap in lexical key order**. This works for: `{datastore: ds-1}` → "ds-1", `{datastoreCluster: dsc-gold}` → "dsc-gold", `{tagCategory: tier, tag: gold}` → "gold" (because BTreeMap sorts by key, and `tag` < `tagCategory`). If no Provider mapping exists at all the backend_id falls back to the class name itself. The provider's reconciler (Phase 1B+) is the consumer; vSphere takes a single `datastore: String` so this rule produces what it needs.
- **InfraMachineRead trait pattern (2026-05-26):** `crates/banlieue-controller/src/reconciler/status_mirror::InfraMachineRead` accessors (`initialization`, `addresses`, `failure_domain`, `provider_id`, `conditions`) keep the reconciler decoupled from provider-specific types. Phase 1C / 1D add `impl InfraMachineRead for ProxmoxMachine` / `LibvirtMachine` without touching the reconciler. The impl for `VSphereMachine` uses a `static NO_INIT: InitializationStatus` fallback for the pre-status case.
- **VirtualMachine status-mirror aggregate Ready rule:** `Ready=True` iff `Scheduled=True && (PlacementValid != False) && InfrastructureReady=True`. Reasons when False: `Scheduling` (not scheduled), `PlacementInvalid` (current placement doesn't satisfy spec), `InfrastructureNotReady` (infra not ready), `Reconciled` (success). The `PlacementValid` condition is treated as "valid unless explicitly False" so we don't require it to exist before iteration 3 lands the migration loop.
- **Infra CR naming convention (2026-05-26):** the owned `VSphereMachine` shares **name + namespace** with the parent `VirtualMachine`. This is the convention for 1:1 owned infra CRs — keeps the relationship discoverable via `kubectl get vspheremachine <vm-name>` without needing an index. Owner-reference back to the VM has `controller: true, blockOwnerDeletion: true`.
- **kube field manager strings are unique per controller (locked):** `banlieue.io/controller` (main controller writes to VMs + InfraMachines), `banlieue.io/provider-vsphere` (vSphere provider writes to VSphereMachine status + VMImage perProvider status + Provider status). Two managers writing the same field cause SSA to fight unless one yields ownership — never reuse a field manager string across binaries.
- **Smoke-test boundary at iteration 2 (2026-05-26):** the controller now reconciles end-to-end against the kind cluster but stops at `Scheduled=False reason=ImageNotReady` because no provider populates `VMImage.status.perProvider[].ready=true`. - **Leader election shape (Phase 1A iter 4, 2026-05-26):** rolled in `banlieue-provider-sdk/src/leader.rs` rather than pulling a third-party crate — `kube-runtime` 3.1 has no built-in. Pure decision function `decide_action(now: Timestamp, lease: Option<&Lease>, cfg) -> LeaseAction { AcquireNew | Renew | Wait | TakeOver }` is unit-tested without a cluster; async `acquire_or_wait` + `renew_forever` are thin wrappers around `Api<Lease>` GET / CREATE / SSA PATCH. Field manager: `banlieue.io/leader-election`. Boundary semantics: `renew_time + lease_duration == now` is still alive (`<=`, not `<`) to avoid racing the holder's renewal cycle. Defaults match `kube-controller-manager`: 15s lease, 5s renew, 2s retry. Identity sourced from `POD_NAME` → `HOSTNAME` → "unknown".
- **Controller startup sequence (post-iter-4):** parse CLI → init tracing → `build_client` → spawn health server → `acquire_or_wait` (unless `--no-leader-elect`) → spawn `renew_forever` background task → build `Controller` → race controller stream against `shutdown_signal()` (SIGTERM + Ctrl-C) via `tokio::select!`. Renew failure inside the background task calls `std::process::exit(1)` so the Deployment restarts the pod and re-enters leader election cleanly.
- **MicroTime is *not* Copy in k8s-openapi 0.27 even when its inner `jiff::Timestamp` is.** Constructing two `MicroTime` values from one `Timestamp::now()` looks like a Copy move and is rejected; the fix is to hold a `let now = Timestamp::now();` and wrap it twice (`MicroTime(now)`, `MicroTime(now)`) — `Timestamp` itself is Copy. Same gotcha will surface when constructing acquire_time/renew_time pairs.
- **TDD test-file `use` path with `#[path]`-included modules:** when `src/foo.rs` declares `#[cfg(test)] #[path = "foo_tests.rs"] mod foo_tests;`, the inner `mod tests` inside `foo_tests.rs` sees `super::super` as `crate::foo` (the parent file's namespace) — **not** `crate::foo::foo_tests` or `crate`. So tests import via `use super::super::{Type, fn};`, NOT `use super::super::foo::{...}`. Got this wrong on the leader_tests and provider_tests first cuts.

Inventory walk pattern: `ViewManager::new(client, &sc.view_manager.value).create_container_view(&sc.root_folder, Some(&["Datacenter".into()]), true).await` → `ContainerView::new(client, &view_ref.value).view().await` returns `Option<Vec<ManagedObjectReference>>`. Then per-moref `Datacenter::new(client, &moref.value).name().await`. **Destroy ContainerViews eagerly** with `view.destroy_view().await` — ignore destroy errors but don't skip the call; vCenter accumulates ghost views otherwise. - **VSphereClient trait pattern (Phase 1B iter 1):** reconcilers depend on `crate::client::VSphereClient` (trait) + `VSphereClientFactory` (builds the client from `ProviderConnection + Credentials`). Even with the trait imported, clippy flags the import as "unused" if the only "use" is implicit coercion. Workaround: add a one-line `fn as_client(c: &FakeClient) -> &dyn VSphereClient { c }` helper that makes the trait usage explicit, then call `discover_inventory(as_client(&client), ...)`. Avoids both the `E0282` "cannot infer type" cascade and the `-D unused-imports` clippy lint.

Per-row reasons can be free-form `String` because they're per-row diagnostics; aggregate-level reasons stay enum-like.

## Do-Not-Repeat

<!-- Mistakes made and corrected. Each entry prevents the same mistake recurring. -->
<!-- Format: [YYYY-MM-DD] Description of what went wrong and what to do instead. -->

- **[2026-05-26] YAML 1.1 implicit-boolean trap in CRD enum variants.** Go's YAML 1.1 parser (used by the kube apiserver) interprets bare `On`/`Off`/`Yes`/`No`/`True`/`False`/`y`/`n` (any case) as booleans. If a CRD enum variant generates one of those bare tokens in its `enum:` or `default:` keys, the apiserver rejects the CRD with `Invalid value: "boolean": ... must be of type string`. **Fix:** never use these names as variant identifiers for types that end up in a CRD schema. Use `PoweredOn`/`PoweredOff`/`Suspended` (vSphere/CAPI convention) or any other multi-word form. Verified 2026-05-26: PowerState was the only offender; all other enums (Firmware kebab-case, IpamSource camelCase, etc.) are safe.
- **[2026-05-26] `calm template --clear-output-directory` clobbers hand-written files in the output dir.** The original `calm-diagrams` Makefile target passed `--clear-output-directory` against `docs/src/architecture/`, which is the same directory MkDocs serves to readers. Adding a hand-written `index.md` (or any other companion doc) to that directory means it gets silently deleted on every `make calm-diagrams` / `make docs` run. **Fix:** replace the blanket clear with an explicit `rm -f $(CALM_DIAGRAMS_OUT)/{system,flows}.md $(CALM_DIAGRAMS_OUT)/*.hbs` of only the files the CLI is about to (re)produce. Applied to both `calm-diagrams` and the new `calm-docify` targets.

## Decision Log

<!-- Significant technical decisions with rationale. Why X was chosen over Y. -->

- **[2026-05-22] CRDs are code-first via `banlieue-api`.** No hand-written CRD YAML. The `crdgen` binary is the only producer of `deploy/crds/`. Rationale: prevents drift between Rust types and deployed schemas (which silently breaks server-side apply).
- **[2026-05-22] Provider communication is CRD-only.** Explicitly rejected gRPC/HTTP between main controller and providers — the K8s API is the bus. Rationale: makes providers reusable as CAPI infra providers, removes a whole class of auth/networking/version-skew problems. See `~/dev/roadmaps/banlieue/00-OVERVIEW.md` (out-of-repo).
- **[2026-05-25] Multi-crate workspace, not single-crate.** Explicitly considered flattening to a single crate like `kube-rs/controller-rs`. Rejected because the roadmap requires 4+ provider crates that ship as independent images. Adopt controller-rs *per-crate conventions* (Error type, telemetry, metrics, fixtures.rs) when each crate lands; don't preload them now. **(Refined 2026-05-30 by ADR-0004 — see below: crates stay independent, but now ship as ONE binary, not independent images.)**
- **[2026-05-30] Single `banlieue` binary with subcommand dispatch (ADR-0004).** Refines the 2026-05-25 multi-crate decision. The role crates (`banlieue-controller`, `banlieue-provider-*`) are now **library-only**; a new thin `crates/banlieue` aggregator is the *only* binary and dispatches `banlieue controller` / `banlieue provider <name>` into them. Per-provider Cargo features (`default = ["vsphere"]`) keep heavy deps (e.g. `vim_rs`) out of slim builds. One image; the Deployment picks the role via container `args` (`["controller"]`, `["provider","vsphere"]`). Rejected: (a) one-binary-per-role (image sprawl + duplicated bootstrap), (b) a single monolithic crate (would force the controller to compile `vim_rs` and collapse the CRD-only seam). Shared bootstrap (tracing/health/shutdown) moved to `banlieue-provider-sdk::bootstrap`. Crate independence + the CRD-only seam are preserved; adding a provider = a feature + a nested subcommand, not a new binary.
- **[2026-05-25] Edition 2024, MSRV 1.85.** Bumped from `2021` / `1.80` to match controller-rs. Rationale: gives access to async closures, new capture rules, and stays aligned with the kube-rs reference. Toolchain check is the only cost.
- **[2026-05-30] CAPI-native cluster provisioning; no native cluster/tier abstraction (ADR-0001).** Considered a banlieue-native `VMTier`/`VMCluster` CRD that fans a tier (platinum=6/6 FDs, gold=5/6, …) out into N VirtualMachines. Rejected: that re-implements CAPI MachineSet/control-plane (replica reconciliation, FD spread, rolling upgrades). banlieue is instead a **CAPI infrastructure provider** — cluster lifecycle is CAPI + a control-plane provider (k0smotron for k0s). "platinum = 6/6" is `replicas: 6` over a cluster advertising 6 FDs. Generic: works with any CAPI consumer, not just k0s. `VirtualMachine` stays CAPI-independent (non-negotiable #3); the CAPI path and standalone-VM path are parallel.
- **[2026-05-30] Add InfraCluster contract via `VSphereCluster` (ADR-0002).** The missing piece for CAPI cluster-side FD spread — banlieue had InfraMachine (`VSphereMachine`) but no InfraCluster. Per-class name (parallels `VSphereMachine`; group disambiguates from CAPV). Rejected a generic `BanlieueCluster` (InfraMachineTemplate is already per-class) and per-Provider InfraCluster (would split a multi-vCenter cluster into multiple CAPI clusters). Topology decision for provider Deployments (per-class vs per-instance vs hybrid, O-003) captured as **Proposed** ADR-0003, not yet locked.
- **[2026-05-25] Roadmap lives outside the repo.** Moved `docs/roadmap/` to `~/dev/roadmaps/banlieue/` and stripped references from in-repo files (`CLAUDE.md`, `.claude/CLAUDE.md`, `.claude/SKILL.md`, `.github/workflows/build.yaml`, `.wolf/cerebrum.md`). Rationale: OSS projects shouldn't ship the maintainer's planning artifacts. ADRs (`docs/adr/`) and design docs (`docs/design/`) **do** stay in-repo — those are public-facing technical records. Roadmap docs follow the same numeric-prefix convention out of tree.

### Do-Not-Repeat (2026-06-01)
- DON'T claim a dependency's TLS backend from a vendored working tree — that tree
  may be PATCHED. Unpatched vim_rs 0.4.4 uses reqwest default-tls → OpenSSL;
  `patches/vim_rs.patch` (= upstream noclue/vim_rs#37) is what switches it to
  rustls. Verify with `git show <REF>:vim_rs/Cargo.toml` + `cargo tree -i openssl-sys`.
- Cargo features are ADDITIVE across the graph: a downstream crate choosing rustls
  CANNOT subtract OpenSSL while an upstream dep keeps default-tls. BYOC
  (injecting a reqwest::Client) controls RUNTIME policy only, NOT which TLS
  backend is compiled in. Keep "who owns the client" and "which TLS backend"
  as separate decisions.

### Key Learning (2026-06-01)
- vim_rs supports BYOC natively: `ClientBuilder::http_client(reqwest::Client)`
  (core/client.rs). `http_client()` and `insecure()` are mutually exclusive
  (each resets the other). Use BYOC to honour Provider.connection.caBundle —
  which is currently DEAD CONFIG (only referenced in a doc comment, never wired).
- noclue/vim_rs#37 = "Make the TLS backend selectable" — strictly TLS backend,
  no behavioural/vcsim/cert changes. When it ships: bump dep, delete patch +
  vendoring pipeline (see patches/README.md "Retiring the patch").

### Key Learning (2026-06-03)
- vim_rs 0.5 is on crates.io: reqwest 0.13/rustls, BYOC via `default-client`
  feature OFF → `ClientBuilder::new(server, client)` 2-arg (NO `.http_client()`).
  The entire 0.4.x vendor/patch/[patch.crates-io] pipeline is DELETED — plain
  `vim_rs = "0.5"` dep. No more `make vendor-vim-rs`.
- reqwest 0.13 `rustls-no-provider` requires a process-default rustls
  CryptoProvider or it PANICS "No provider set" at first TLS use. We install ring
  (rustls::crypto::ring::default_provider().install_default(), idempotent) once at
  provider run() startup. ring is shared with kube → single crypto provider, no
  aws-lc-rs, no OpenSSL. Choosing reqwest's plain `rustls` feature instead would
  pull aws-lc-rs (C/asm, cross-compile friction).
